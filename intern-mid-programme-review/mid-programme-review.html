2. Performance Review — Query Tuning

The Problem: Regex on Unindexed Fields

The current free-text search in GET /api/candidates uses a $regex across five fields. On a small seed dataset this is invisible, but it does a full collection scan on every request — MongoDB cannot use any index for a leading-wildcard regex.

// Current code — index.js line 57-64
// ❌ Five separate $regex expressions — full collection scan every time
const textMatch = q.trim()
  ? {
      $or: [
        { name:     { $regex: q, $options: 'i' } },
        { bio:      { $regex: q, $options: 'i' } },
        { skills:   { $regex: q, $options: 'i' } },
        { location: { $regex: q, $options: 'i' } },
        { role:     { $regex: q, $options: 'i' } },
      ],
    }
  : {};
Why this hurts at scale: With 100,000 candidates every search reads every document in the collection — O(n). Response times go from ~5 ms (100 docs) to ~4 s (100 k docs) with no code change.

Fix 1 — MongoDB Atlas Search (recommended for production)

If you use MongoDB Atlas, replace the $regex with a $search stage. Atlas maintains a Lucene index — same technology as Elasticsearch.

// ✅ Atlas full-text search (replace textMatch block)
const textMatch = q.trim()
  ? { $text: { $search: q } }   // requires a text index — see section 3
  : {};

Fix 2 — MongoDB text index (self-hosted / local)

Create a compound text index once (see section 3). Then replace all five $regex expressions with a single $text operator. MongoDB scores each document and only touches the index, not the full documents.

// ✅ Single $text operator — uses the text index created in section 3
const textMatch = q.trim()
  ? { $text: { $search: q } }
  : {};

Fix 3 — Filter-field indexes (skills, location, status)

The $match inside the $facet pipeline also runs without an index. Each filter field should have its own index.

// Run once in mongo shell or a migration script
db.candidates.createIndex({ skills: 1 });
db.candidates.createIndex({ location: 1 });
db.candidates.createIndex({ status: 1 });
Analogy for interns: Imagine a library with no catalogue. A full-collection-scan is a librarian who reads every book title to find the one you want. An index is the catalogue — the librarian goes straight to the shelf.

How to Verify the Fix with explain()

// In Mongosh — check the winning plan before and after adding indexes
db.candidates.find({ $text: { $search: "React" } }).explain("executionStats")

// Look for:
//   "winningPlan.stage": "TEXT"   ← good — using text index
//   "winningPlan.stage": "COLLSCAN" ← bad — full scan, add the index

3. Index Mapping Optimisation

Current State — No Explicit Indexes

The Candidate Mongoose schema has no index declarations beyond the default _id. Every sort, filter, and text-search stage does a collection scan.

Recommended Index Strategy

Index Type Used by Why needed
{ name, bio, skills, location, role } Text (compound) GET ?q= free-text Replaces 5 regex scans with one scored lookup
{ skills: 1 } Multikey GET ?skills= filter Skills is an array — MongoDB builds a multikey index automatically
{ location: 1 } Single-field GET ?locations= filter Equality match on location facet
{ status: 1 } Single-field GET ?statuses= filter Low cardinality — still worth indexing for large collections
{ experience_years: -1 } Single-field GET ?sort=exp_desc Without this, the sort stage buffers all documents in memory
{ email: 1 } Unique sparse POST /api/candidates Prevents duplicate email entries at the DB layer

Adding Indexes in the Mongoose Schema

The best place to declare indexes is in models/Candidate.js — they are created automatically when the app connects for the first time.

// models/Candidate.js — optimised version
const mongoose = require('mongoose');

const candidateSchema = new mongoose.Schema(
  {
    name:             { type: String, required: true, trim: true },
    email:            { type: String, trim: true, lowercase: true },
    phone:            { type: String, trim: true },
    role:             { type: String, trim: true },
    status:           {
      type: String,
      enum: ['Applied', 'Interview', 'Hired', 'Rejected'],
      default: 'Applied',
      index: true,           // ← single-field index
    },
    skills:           [{ type: String, trim: true }],   // multikey index below
    location:         { type: String, trim: true, index: true },
    experience_years: { type: Number, default: 0 },
    bio:              { type: String, trim: true },
    applied_at:       { type: Date, default: Date.now },
  },
  { timestamps: true }
);

// Compound text index — covers all free-text search fields
candidateSchema.index(
  { name: 'text', bio: 'text', skills: 'text', location: 'text', role: 'text' },
  { weights: { name: 10, skills: 5, role: 5, bio: 2, location: 1 } }
);

// Multikey index on skills array (for facet filter)
candidateSchema.index({ skills: 1 });

// Sort support
candidateSchema.index({ experience_years: -1 });

// Unique email (sparse = allows multiple null/missing emails)
candidateSchema.index({ email: 1 }, { unique: true, sparse: true });

module.exports = mongoose.model('Candidate', candidateSchema);
Weight tip: The text index weights mean a match in name scores 10× higher than a match in location. This gives more intuitive search ranking with zero extra code.
Watch out: MongoDB only allows one text index per collection. If you call createIndex twice with different text fields you will get an error. Declare all text fields in one compound text index.

4. Writing Unit Tests for API Routes (Jest)

Setup

We use Jest as the test runner and Supertest to make real HTTP calls against the Express app without starting a server. mongodb-memory-server spins up an in-memory MongoDB so tests are fast and isolated.

# Install test dependencies (run from /server)
npm install --save-dev jest supertest mongodb-memory-server @types/jest

Add the following to package.json:

{
  "scripts": {
    "test": "jest --runInBand --forceExit"
  },
  "jest": {
    "testEnvironment": "node",
    "testMatch": ["**/__tests__/**/*.test.js"]
  }
}

 * __tests__/candidates.test.js
 * Unit tests for GET /api/candidates and POST /api/candidates

const request  = require('supertest');
const mongoose = require('mongoose');
const { MongoMemoryServer } = require('mongodb-memory-server');

// We import the app but NOT the listen() call — supertest handles the port
let app;
let mongod;

// ── Lifecycle ────────────────────────────────────────────────────────────────
beforeAll(async () => {
  mongod = await MongoMemoryServer.create();
  const uri = mongod.getUri();

  // Override the connection string before requiring the app
  process.env.MONGO_URI = uri;

  // Require app after env is set so it connects to in-memory DB
  app = require('../app');   // NOTE: you need to split app.js from index.js (see below)
  await mongoose.connect(uri);
});

afterEach(async () => {
  // Clear all collections between tests so tests don't bleed into each other
  const collections = mongoose.connection.collections;
  for (const key in collections) {
    await collections[key].deleteMany({});
  }
});

afterAll(async () => {
  await mongoose.disconnect();
  await mongod.stop();
});

#FindingSeverityWhat to do

A Code Smell to Discuss

Look at this line in the facet pipeline:

// index.js line 101 — location_facet
{ $match: { location: { $exists: true, $ne: null, $ne: '' } } },
Duplicate key issue: The object literal has $ne twice — the second one silently overwrites the first in JavaScript. This means you're only guarding against "" (empty string) but not null. Fix:

{ $match: { location: { $exists: true, $nin: [null, ''] } } }
