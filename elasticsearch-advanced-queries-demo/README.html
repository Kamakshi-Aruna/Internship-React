Topic 1

1. match Query — Full-Text Search on One Field

match is the standard query for full-text search. It analyzes your search text (lowercases it, splits into words) and finds documents that contain those words.

Think of it like: Google search — you type "react developer" and it finds all pages that mention "react" or "developer", ranked by relevance.

Basic match — find candidates who know React:

GET /candidates/_search
{
  "query": {
    "match": {
      "skills": "React"
    }
  }
}
Returns: Alice, Carol, David, Frank — all have React in skills

match with multiple words — OR behavior by default:

GET /candidates/_search
{
  "query": {
    "match": {
      "skills": "React Python"
    }
  }
}
Returns: Anyone with React OR Python — Alice, Carol, David, Frank, and Carol/Frank who have Python too. Documents with BOTH score higher.

match with AND operator — must contain ALL words:

GET /candidates/_search
{
  "query": {
    "match": {
      "skills": {
        "query": "React Python",
        "operator": "and"
      }
    }
  }
}
Returns: Only Carol and Frank — they have BOTH React AND Python

match on bio field — search inside text:

GET /candidates/_search
{
  "query": {
    "match": {
      "bio": "full-stack developer"
    }
  }
}
OptionDefaultWhat it does
operator: "or"YesMatch any word — more results
operator: "and"NoMatch all words — stricter results
fuzziness: "AUTO"NoAllow typos — "Raect" still finds "React"

Bonus — match with fuzziness (typo tolerance):

GET /candidates/_search
{
  "query": {
    "match": {
      "skills": {
        "query": "Raect",
        "fuzziness": "AUTO"
      }
    }
  }
}
"Raect" still finds "React" — useful for search boxes where users make typos
Topic 2

2. multi_match Query — Search Across Multiple Fields

multi_match runs the same search query across multiple fields at once. Instead of searching only skills, you can search skills, name, and bio simultaneously.

Think of it like: Searching a resume — you want "React" to match whether it appears in the skills section, the job title, or the description paragraph.

Search "React" across skills AND bio:

GET /candidates/_search
{
  "query": {
    "multi_match": {
      "query": "React developer",
      "fields": ["skills", "bio", "name"]
    }
  }
}

Boost a specific field — skills match counts more than bio:

GET /candidates/_search
{
  "query": {
    "multi_match": {
      "query": "React",
      "fields": ["skills^3", "bio^1", "name^2"]
    }
  }
}
What does ^3 mean? It's a boost multiplier.
skills^3 = a match in skills is 3x more important than a match in bio.
This affects the _score — candidates with React in their skills rank higher than those who only mention it in their bio.

multi_match types — control HOW fields are searched:

GET /candidates/_search
{
  "query": {
    "multi_match": {
      "query": "React developer New York",
      "fields": ["skills", "bio", "location"],
      "type": "best_fields"
    }
  }
}
typeBehaviorBest for
best_fieldsUses the score of the single best matching fieldShort text like name/title (default)
most_fieldsAdds up scores from all matching fieldsSame content indexed multiple ways
cross_fieldsTreats all fields as one big field"New York" split across city + state fields
phraseWords must appear in exact orderExact phrase matching
Topic 3

3. bool Query — Combine Multiple Conditions

The bool query is the most powerful query in Elasticsearch. It lets you combine any number of queries using four clauses:

ClauseMust match?Affects score?SQL equivalent
mustYes — requiredYes — boosts scoreAND (with ranking)
filterYes — requiredNo — just yes/noWHERE (fast)
shouldNo — optionalYes — if matches, boosts scoreOR (nice to have)
must_notMust NOT matchNoAND NOT
Simple rule:
Use must for full-text search (relevance matters)
Use filter for yes/no conditions like status, location, date (no scoring needed — faster)

Example 1 — must + filter (most common pattern):

Find active candidates who know React with 3+ years experience:

GET /candidates/_search
{
  "query": {
    "bool": {
      "must": [
        { "match": { "skills": "React" } }
      ],
      "filter": [
        { "term":  { "status": "active" } },
        { "range": { "experience_years": { "gte": 3 } } }
      ]
    }
  }
}
Returns: Alice (4yr, active, React) and Frank (7yr, active, React) — NOT David (inactive) or Carol (no React in top match)

Example 2 — should (boost if matches, not required):

Find active candidates — but rank those in New York higher:

GET /candidates/_search
{
  "query": {
    "bool": {
      "filter": [
        { "term": { "status": "active" } }
      ],
      "should": [
        { "term": { "location": "New York" } }
      ]
    }
  }
}
Returns: All active candidates, but New York candidates appear at the top with a higher _score

Example 3 — must_not (exclude results):

Find all candidates who are NOT inactive:

GET /candidates/_search
{
  "query": {
    "bool": {
      "must_not": [
        { "term": { "status": "inactive" } }
      ]
    }
  }
}

Example 4 — all four clauses together:

GET /candidates/_search
{
  "query": {
    "bool": {
      "must": [
        { "match": { "bio": "developer" } }
      ],
      "filter": [
        { "term":  { "status": "active" } },
        { "range": { "experience_years": { "gte": 2 } } }
      ],
      "should": [
        { "term": { "location": "New York" } },
        { "match": { "skills": "React" } }
      ],
      "must_not": [
        { "term": { "location": "Austin" } }
      ]
    }
  }
}
minimum_should_match: By default if you have only should clauses (no must/filter), at least 1 must match. You can control this:
"minimum_should_match": 2 — at least 2 should clauses must match
Topic 4

4. Filters — term, range, exists

Filters are yes/no checks — they don't affect relevance score, they just include or exclude documents. Always put filters inside "filter": [] inside a bool query for best performance.

4.1 — term filter (exact match)

Use for keyword fields — status, location, email. Must be exact, case-sensitive.

term Find all active candidates
GET /candidates/_search
{
  "query": {
    "bool": {
      "filter": [
        { "term": { "status": "active" } }
      ]
    }
  }
}
terms Match multiple values (IN list)
GET /candidates/_search
{
  "query": {
    "bool": {
      "filter": [
        { "terms": { "location": ["New York", "Austin"] } }
      ]
    }
  }
}

Returns candidates in New York OR Austin — like SQL WHERE location IN ('New York', 'Austin')

Wrong — term on text field

{ "term": { "skills": "React" } }
// skills is a TEXT field — term
// won't work reliably here

Correct — term on keyword field

{ "term": { "skills.keyword": "React" } }
// use .keyword sub-field for exact match
// OR use match for text fields

4.2 — range filter (numbers and dates)

range Experience between 3 and 6 years
GET /candidates/_search
{
  "query": {
    "bool": {
      "filter": [
        {
          "range": {
            "experience_years": {
              "gte": 3,
              "lte": 6
            }
          }
        }
      ]
    }
  }
}
OperatorMeaningExample
gtegreater than or equal"gte": 3 → 3, 4, 5...
gtgreater than (exclusive)"gt": 3 → 4, 5...
lteless than or equal"lte": 6 → ...4, 5, 6
ltless than (exclusive)"lt": 6 → ...4, 5

4.3 — exists filter (field has a value)

Check if a field exists and has a non-null value in the document.

exists Find candidates who have a bio
GET /candidates/_search
{
  "query": {
    "bool": {
      "filter": [
        { "exists": { "field": "bio" } }
      ]
    }
  }
}
must_not + exists Find candidates who do NOT have a bio
GET /candidates/_search
{
  "query": {
    "bool": {
      "must_not": [
        { "exists": { "field": "bio" } }
      ]
    }
  }
}
exists is useful when: Some documents are missing optional fields. For example — candidates who haven't filled in their bio, or jobs without a salary field.
Topic 5

5. Highlight — Show Matched Terms in Results

Highlight tells Elasticsearch to return snippets of text where the search term was found, with the matching word wrapped in <em> tags (or custom tags). This is how search engines show you bold matched words in results.

Think of it like: When you Google something, the result description shows your search words in bold. That's exactly what highlight does in Elasticsearch.

Basic highlight on bio field:

GET /candidates/_search
{
  "query": {
    "match": {
      "bio": "React developer"
    }
  },
  "highlight": {
    "fields": {
      "bio": {}
    }
  }
}

What the response looks like:

{ "_source": { "name": "Alice Johnson", "bio": "Alice is an experienced React developer who loves building fast user interfaces." }, "highlight": { "bio": [ "Alice is an experienced <em>React</em> <em>developer</em> who loves building fast user interfaces." ] } }
In your React frontend: Use dangerouslySetInnerHTML to render the highlight snippet — the <em> tags will make matched words italic/bold.

Custom highlight tags (use <strong> instead of <em>):

GET /candidates/_search
{
  "query": {
    "match": { "bio": "React" }
  },
  "highlight": {
    "pre_tags":  ["<strong>"],
    "post_tags": ["</strong>"],
    "fields": {
      "bio": {}
    }
  }
}

Highlight multiple fields at once:

GET /candidates/_search
{
  "query": {
    "multi_match": {
      "query": "React developer",
      "fields": ["bio", "skills"]
    }
  },
  "highlight": {
    "fields": {
      "bio":    {},
      "skills": {}
    }
  }
}

Control snippet size (number of characters shown):

GET /candidates/_search
{
  "query": {
    "match": { "bio": "React" }
  },
  "highlight": {
    "fields": {
      "bio": {
        "fragment_size": 80,
        "number_of_fragments": 2
      }
    }
  }
}
OptionDefaultWhat it does
fragment_size100Characters in each snippet
number_of_fragments5How many snippets to return per field
pre_tags<em>Opening tag around matched word
post_tags</em>Closing tag around matched word
Topic 6

6. Combined — Full-Text Search + Structured Filters + Highlight

This is how a real talent search feature works — combining everything you learned into one powerful query.

Real-world example: "Find active React developers in New York with 3+ years, highlight matched bio text"

GET /candidates/_search
{
  "query": {
    "bool": {
      "must": [
        {
          "multi_match": {
            "query": "React developer",
            "fields": ["skills^3", "bio^1"],
            "type": "best_fields"
          }
        }
      ],
      "filter": [
        { "term":  { "status": "active" } },
        { "term":  { "location": "New York" } },
        { "range": { "experience_years": { "gte": 3 } } },
        { "exists": { "field": "bio" } }
      ]
    }
  },
  "highlight": {
    "fields": {
      "bio":    { "fragment_size": 100 },
      "skills": {}
    }
  },
  "sort": [
    { "_score": { "order": "desc" } },
    { "experience_years": { "order": "desc" } }
  ],
  "_source": ["name", "email", "skills", "location", "experience_years", "status"]
}

Breaking down each part:

PartWhat it does
must → multi_matchFull-text search "React developer" across skills (3x boost) and bio — affects ranking
filter → term statusOnly active candidates — hard yes/no, no scoring
filter → term locationOnly New York candidates
filter → range experienceOnly 3+ years experience
filter → exists bioOnly candidates who have a bio
highlightShow matched words in bio and skills with <em> tags
sortSort by relevance score first, then experience
_sourceOnly return these fields (like SQL SELECT — skip bio from source since highlight covers it)

Simpler combined query — search bar with filters (typical search page):

GET /candidates/_search
{
  "query": {
    "bool": {
      "must": [
        { "match": { "skills": "Python" } }
      ],
      "filter": [
        { "term":  { "status": "active" } },
        { "range": { "experience_years": { "gte": 2, "lte": 8 } } }
      ],
      "should": [
        { "term": { "location": "New York" } }
      ]
    }
  },
  "highlight": {
    "fields": { "skills": {}, "bio": {} }
  }
}

Golden Rule:

must vs filter — when to use which?

must → when relevance/ranking matters (full-text search fields like skills, bio, name)
filter → when it's a hard yes/no condition (status, location, date, number ranges)

Filters are cached by Elasticsearch and are significantly faster — always prefer filter for structured conditions.
