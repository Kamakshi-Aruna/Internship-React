Overview

1. The Big Picture — How the App Works

Talent Tracker is a full-stack web application for managing job candidates and job postings. Think of it as a simple version of a hiring platform — you can add candidates, track their status (Applied → Interview → Hired), and manage job openings.

🖥️

Frontend (Client)

React app the user sees in the browser — pages, buttons, tables, charts

⚙️

Backend (Server)

Node.js + Express — receives API requests, talks to the database

🗄️

Database

MongoDB — stores all candidate and job data permanently

🔐

Authentication

Firebase — handles login/logout so only authorised users can access the app

Simple Analogy: Think of a restaurant. The Frontend is the dining area (what customers see). The Backend is the kitchen (processes requests). The Database is the fridge/pantry (stores all the data). Firebase is the bouncer at the door (only lets logged-in users in).

4. Server — Entry Point & Config

🚀 server/index.js
Entry Point

What is it? The very first file Node.js runs when you type npm start in the server folder. It boots up the entire backend.

Why do we need it? It stitches everything together — connects to MongoDB, registers all the route files, adds security headers, and starts listening for incoming requests on port 5000.

// What index.js does, step by step:
const app = express();         // 1. Create an Express "app" (a web server)
app.use(helmet());             // 2. Add security headers automatically
app.use(cors());               // 3. Allow the React frontend to talk to this server
app.use(express.json());       // 4. Allow receiving JSON in request bodies

app.use('/api/candidates', candidateRoutes); // 5. Register route files
app.use('/api/jobs',        jobRoutes);
app.use('/api/stats',       statsRoutes);

app.listen(5000, ...);         // 6. Start listening on port 5000
Analogy: Think of this file as the manager who opens the restaurant, unlocks the door, assigns staff to their stations, and turns on the "OPEN" sign.
📦 server/package.json
Config

What is it? A list of every external library (npm package) the server needs to run.

Why do we need it? When you run npm install, npm reads this file and downloads all the listed libraries automatically. Without it, you'd have to install each library by hand.

// Key packages in server/package.json:
"express"         — the web server framework
"mongoose"        — talks to MongoDB (adds nice JS methods)
"firebase-admin"  — verifies Firebase login tokens on the server
"helmet"          — adds security headers to every response
"cors"            — allows the React app to call this API
"morgan"          — logs every incoming request (great for debugging)
"dotenv"          — loads secret values from the .env file
🔑 server/.env   &   server/.env.example
Secret Config

What is it? A file that stores sensitive values like database passwords and API keys. These values change between environments (local laptop vs. production server).

Why do we need it? We never put passwords directly in code because the code goes into Git (which is public). The .env file is listed in .gitignore so it never gets uploaded.

# server/.env (never commit this file!)
PORT=5000
MONGO_URI=mongodb://localhost:27017/talent-tracker
FIREBASE_PROJECT_ID=your-firebase-project-id
FIREBASE_CLIENT_EMAIL=firebase-adminsdk@your-project.iam.gserviceaccount.com

Important: .env.example is the safe version — it shows which keys are needed but leaves the values blank. New developers copy this file, rename it to .env, and fill in their own values.

What is it? A small file whose only job is to connect the server to MongoDB.

Why do we need it? We separate the database connection into its own file so it can be imported once in index.js and reused. If the DB URL changes we only need to update one place.

Analogy: This is like the water pipe connection to the restaurant kitchen. Without it, nothing works — all the cooking (data operations) needs water (the database).

5. Server — Middleware

What is middleware? Middleware is code that runs between a request arriving and your route handler responding. Think of it as a series of security checks and processors a request passes through before reaching its destination.
🔒 server/middleware/authMiddleware.js
Authentication

What is it? A gatekeeper function that checks every API request to make sure the user is logged in before letting them through.

Why do we need it? Without this, anyone in the world could send a request to /api/candidates and read or delete your data. This file checks the Firebase login token that the frontend sends with every request.

// How it works (simplified):
const protect = async (req, res, next) => {
  // 1. Get the token from the request header: "Authorization: Bearer abc123..."
  const token = req.headers.authorization?.split(' ')[1];

  if (!token) return res.status(401).json({ message: 'No token' });

  // 2. Ask Firebase: "Is this a real, valid token?"
  const decoded = await admin.auth().verifyIdToken(token);

  // 3. If yes, attach the user info and let the request continue
  req.user = { uid: decoded.uid, email: decoded.email };
  next();  // ← "next" means "pass this request to the actual route handler"
};
Analogy: This is the bouncer at the door. Every person (request) must show a valid ID (Firebase token). No ID = no entry.
🩹 server/middleware/errorHandler.js
Error Handling

What is it? A central place that catches all errors from any route and sends back a clean, consistent error response.

Why do we need it? Without it, every route would need its own error handling code (very repetitive). With it, any route can just call next(err) and this file handles the rest. It also translates MongoDB-specific errors into readable messages.

// errorHandler.js — what it catches:

// 1. Mongoose validation error (e.g., name field is required but missing)
//    → returns 400 with "name: Path `name` is required"

// 2. Duplicate key error (e.g., same email used twice)
//    → returns 400 with "Duplicate value: email already exists"

// 3. Invalid MongoDB ID (e.g., malformed _id in the URL)
//    → returns 400 with "Invalid ID format"

// 4. Everything else
//    → returns 500 with the error message

6. Server — Models (Database Schemas)

What is a Model/Schema? A schema defines the shape of your data — which fields exist, what type they are, and which ones are required. MongoDB is flexible by default, but schemas enforce structure. A Model is the JavaScript class you use to create/read/update/delete documents in that collection.
👤 server/models/Candidate.js
Database Schema

What is it? Defines what a "Candidate" document looks like in MongoDB — what fields it has and what rules apply.

Why do we need it? Without it, you could accidentally store a candidate with no name, a typo in status, or a duplicate email. The schema acts as a contract — data must match this shape or it gets rejected.

// server/models/Candidate.js — what a Candidate looks like
{
  name:   String,   // required — must be provided
  email:  String,   // required + unique — no two candidates can share an email
  phone:  String,   // optional
  status: String,   // must be one of: "Applied", "Interview", "Hired", "Rejected"
  role:   String,   // optional — e.g. "Frontend Developer"

  // Auto-added by Mongoose (timestamps: true):
  createdAt: Date,
  updatedAt: Date,
}
Analogy: Imagine a paper form with labelled boxes. The schema is the printed form template — it decides which boxes exist and which ones you must fill in.
💼 server/models/Job.js
Database Schema

What is it? Defines what a "Job" document looks like in MongoDB.

Why do we need it? Same reason as Candidate — enforces data rules so no invalid job records get saved.

// server/models/Job.js — what a Job looks like
{
  title:      String,   // required — e.g. "Senior React Developer"
  department: String,   // optional — e.g. "Engineering"
  location:   String,   // optional — e.g. "Remote"
  status:     String,   // must be: "Open", "Closed", or "On Hold"
  openings:   Number,   // how many people to hire (minimum: 1)

  // Auto-added:
  createdAt: Date,
  updatedAt: Date,
}

7. Server — Routes (API Endpoints)

What is a Route? A route is a URL path + HTTP method that the server "listens" for. When the frontend calls GET /api/candidates, the server runs the matching route handler function and sends back data.
👥 server/routes/candidates.js
API Routes

What is it? Defines the four CRUD (Create, Read, Update, Delete) operations for candidates.

Why do we need it? The frontend needs a way to get candidate data, add new candidates, edit them, and delete them. Each of these actions needs its own URL endpoint. All routes are protected — only logged-in users can use them.

MethodURLWhat it does
GET/api/candidatesFetch all candidates (newest first)
POST/api/candidatesCreate a new candidate
PUT/api/candidates/:idUpdate a specific candidate by ID
DELETE/api/candidates/:idDelete a specific candidate by ID
 
// Example — GET all candidates
router.get('/', protect, async (req, res, next) => {
  try {
    const candidates = await Candidate.find().sort({ createdAt: -1 });
    res.json(candidates);
  } catch (err) {
    next(err);  // passes the error to errorHandler.js
  }
});
💼 server/routes/jobs.js
API Routes

What is it? Same as candidates.js but for Job documents.

Why do we need it? Jobs are a completely separate data type, so they get their own route file. The pattern is identical: GET all, POST create, PUT update, DELETE remove.

MethodURLWhat it does
GET/api/jobsFetch all jobs
POST/api/jobsCreate a new job

📊 server/routes/stats.js
Aggregation

What is it? A special route that runs multiple MongoDB aggregation queries to produce summary statistics for the Dashboard page.

Why do we need it? The Dashboard shows cards like "Total Candidates: 45" and charts like "Applied: 20, Interview: 10, Hired: 15". This data doesn't exist as a single document — it has to be calculated by counting and grouping the records. That is what aggregation does.

// stats.js — runs 5 queries in parallel
const [
  totalCandidates,   // simple count: how many candidates total?
  totalJobs,         // simple count: how many jobs total?
  candidatesByStatus,// group + count: {Applied: 20, Interview: 10, Hired: 15}
  jobsByStatus,      // group + count: {Open: 8, Closed: 3, "On Hold": 2}
  recentCandidates,  // latest 5 candidates added
] = await Promise.all([...]);

res.json({ totalCandidates, totalJobs, candidatesByStatus, jobsByStatus, recentCandidates });
Analogy: Instead of handing you the entire employee spreadsheet, this route is like the HR manager who reads the whole spreadsheet and hands you a one-page summary report.

10. Client — Firebase & API

🔥 client/src/firebase/firebaseConfig.js
Firebase

What is it? Initialises the Firebase SDK using credentials from .env. Exports the auth object used for login/logout.

Why do we need it? Firebase needs to be initialised once before any auth operations can happen. By doing it here and exporting auth, every other file can import it without re-initialising.

// client/src/firebase/firebaseConfig.js
import { initializeApp } from 'firebase/app';
import { getAuth } from 'firebase/auth';

const firebaseConfig = {
  apiKey:    import.meta.env.VITE_FIREBASE_API_KEY,
  authDomain: import.meta.env.VITE_FIREBASE_AUTH_DOMAIN,
  projectId:  import.meta.env.VITE_FIREBASE_PROJECT_ID,
  appId:      import.meta.env.VITE_FIREBASE_APP_ID,
};

const app  = initializeApp(firebaseConfig);
export const auth = getAuth(app);
Analogy: This is like plugging in the security system — you do it once at the start, and then every door in the building can use it.
📡 client/src/api/axiosInstance.js
HTTP Client

What is it? A pre-configured version of Axios (the library used for HTTP requests). It automatically adds the Firebase login token to every API request.

Why do we need it? Every API request to the server must include an Authorization: Bearer <token> header — that is how the server knows you are logged in. Without this file, every page (Candidates.jsx, Jobs.jsx, etc.) would have to manually fetch the token and add the header. This file does it once for everyone.

// axiosInstance.js — what happens on every API request
axiosInstance.interceptors.request.use(async (config) => {
  const user = auth.currentUser;
  if (user) {
    const token = await user.getIdToken();  // get fresh Firebase token
    config.headers.Authorization = `Bearer ${token}`;  // attach it
  }
  return config;
});

// What happens on a 401 (token expired):
axiosInstance.interceptors.response.use(null, async (error) => {
  if (error.response?.status === 401) {
    await signOut(auth);   // log the user out
    window.location.href = '/login';  // redirect to login page
  }
});
Analogy: Imagine every staff member needs to show their ID badge when entering the kitchen. Instead of each person remembering to clip on their badge, there's a turnstile that automatically reads and attaches their badge every time.

11. Client — Auth Context

🌍 client/src/contexts/AuthContext.jsx
State Management

What is it? A React Context that holds the current logged-in user and shares it with every component in the app. Also provides login() and logout() functions.

Why do we need it? Without context, if AppLayout.jsx needs to know the user's email (to show it in the sidebar), it would need the user passed down as a prop through multiple layers. Context makes the user data available everywhere without "prop drilling".

// How it works — simplified
const AuthContext = createContext();

export function AuthProvider({ children }) {
  const [user, setUser] = useState(null);  // null = not logged in

  useEffect(() => {
    // Firebase fires this whenever login state changes
    const unsubscribe = onAuthStateChanged(auth, (firebaseUser) => {
      setUser(firebaseUser);  // update state automatically
    });
    return unsubscribe;  // cleanup on unmount
  }, []);

  const login  = (email, password) => signInWithEmailAndPassword(auth, email, password);
  const logout = () => signOut(auth);

  return <AuthContext.Provider value={{ user, login, logout }}>{children}</AuthContext.Provider>;
}

// Any component can now do:
const { user, logout } = useAuth();
Analogy: Context is like a company-wide announcement board. Instead of one person walking around telling everyone the news, it's posted centrally and anyone who needs it can look it up.

What is it? The main shell of the app — it wraps every protected page with a collapsible sidebar (navigation menu) on the left and a content area on the right.

🛡️ client/src/components/ProtectedRoute.jsx
Route Guard

What is it? A wrapper component that checks if the user is logged in. If yes, it shows the wrapped page. If no, it redirects to /login.

Why do we need it? Without this, anyone could type /candidates in the browser address bar and see all your data without logging in. This component acts as a gate.

// ProtectedRoute.jsx — simplified
function ProtectedRoute({ children }) {
  const { user, loading } = useAuth();

  if (loading) return <Spinner />;  // wait while Firebase checks login state

  if (!user) return <Navigate to="/login" />;  // not logged in → go to login

  return children;  // logged in → show the protected page
}
Analogy: This is the lock on a private room. The key is being logged in. Without the key, the door sends you back to reception (login page).

13. Client — Pages

🔐 client/src/pages/Login.jsx
Page

What is it? The login screen — email + password fields, submit button, and error messages.

Why do we need it? It is the entry point for every user. On successful login, it navigates to /dashboard. It maps Firebase error codes (like auth/wrong-password) to user-friendly messages.

// Login.jsx — key logic
const handleSubmit = async (values) => {
  try {
    await login(values.email, values.password);  // from AuthContext
    navigate('/dashboard');
  } catch (err) {
    // Map Firebase error codes → readable messages
    const messages = {
      'auth/wrong-password': 'Incorrect password.',
      'auth/user-not-found': 'No account with that email.',
    };
    setError(messages[err.code] || 'Login failed.');
  }
};

// What Candidates.jsx does:
// 1. On load: GET /api/candidates → display in AG Grid table

14. How All the Files Connect

Here is the full journey of what happens when a logged-in user opens the Candidates page:

  • 1 User visits /candidates in the browser
  • 2 React Router (in App.jsx) matches the path
  • 3 ProtectedRoute.jsx checks AuthContext — user is logged in, so it proceeds
  • 4 AppLayout.jsx renders the sidebar; Candidates.jsx renders in the content area
  • 5 Candidates.jsx calls axiosInstance.get('/api/candidates')
  • 6 axiosInstance.js fetches the Firebase token and adds it to the request header
  • 7 Request arrives at the Express server (server/index.js)
  • 8 authMiddleware.js verifies the token with Firebase Admin SDK
  • 9 routes/candidates.js runs Candidate.find()
  • 10 Mongoose queries MongoDB using the schema from models/Candidate.js
  • 11 Data flows back: MongoDB → Mongoose → Route → Express → Axios → Candidates.jsx
  • 12 AG Grid renders the candidate table with the returned data
The whole system in one sentence: The pages ask for data, axiosInstance attaches the security badge, the server routes verify the badge and query the database, and the data flows back to fill the table on screen.
