Concept

1. What is Elasticsearch?

Elasticsearch is a search engine database. While MongoDB stores data and lets you filter it, Elasticsearch stores data and lets you search it super fast — even across millions of records, with full-text search, typo tolerance, ranking, and filters combined.

Simple Analogy:
MongoDB is like a filing cabinet — great for storing and retrieving exact records.
Elasticsearch is like Google Search — great for finding relevant results from a huge pile of data, ranked by how well they match your query.

When do you use Elasticsearch in Talent Tracker?

🔍

Full-Text Search

Search "React developer New York" and get best-match candidates

⚡

Fast Autocomplete

Type "Java" and instantly see matching skills/candidates

🎯

Relevance Ranking

Results sorted by how well a candidate matches the job

📊

Aggregations

Count candidates by skill, location, or experience level

Core Concepts

2. Core Concepts — Index, Document, Mapping

2.1 Index

An Index is like a table in a database, or a collection in MongoDB. It holds a group of related documents. In our project we will have an index called candidates.

Think of it this way: MongoDB Collection candidates → Elasticsearch Index candidates

2.2 Document

A Document is a single record inside an index — it is a JSON object. Just like one candidate object in MongoDB is one document, one candidate in Elasticsearch is also one document. 

{
  "_id": "001",
  "_source": {
    "name": "Alice Johnson",
    "email": "alice@example.com",
    "skills": ["React", "Node.js", "MongoDB"],
    "location": "New York",
    "experience_years": 4,
    "status": "active"
  }
}

2.3 Mapping

A Mapping is like a schema — it tells Elasticsearch what type each field is: text, keyword, number, date, boolean, etc. This matters because:

  • text fields are analyzed (broken into words) — used for full-text search
  • keyword fields are stored as-is — used for exact match, sorting, filtering
  • integer / float — used for numeric range queries
  • date — used for date range queries
Tip: If you want to both search AND filter on a field (like skills), use multi-fields: map it as both text (for search) and keyword (for exact filter/aggregation).

Example Mapping for the candidates index:

PUT /candidates
{
  "mappings": {
    "properties": {
      "name": {
        "type": "text",
        "fields": {
          "keyword": { "type": "keyword" }
        }
      },
      "email":            { "type": "keyword" },
      "skills": {
        "type": "text",
        "fields": {
          "keyword": { "type": "keyword" }
        }
      },
      "location":         { "type": "keyword" },
      "experience_years": { "type": "integer" },
      "status":           { "type": "keyword" },
      "applied_at":       { "type": "date" }
    }
  }
}
Field Type Why this type?
nametext + keywordFull-text search + exact sort
emailkeywordAlways exact match, never full-text
skillstext + keywordSearch "react" loosely + filter exact "React"
locationkeywordFilter by exact city name
experience_yearsintegerRange queries like >= 3 years
statuskeywordExact filter: active / inactive
applied_atdateSort and filter by date

Setup

4. Set Up Elasticsearch + Kibana with Docker

We use Docker so you don't need to install anything manually. Everything runs in containers.

Step 1 — Prerequisites

  • Install Docker Desktop and make sure it is running
  • Make sure you have at least 4 GB RAM free (Elasticsearch is memory-hungry)
  • Open your terminal in this folder: elasticsearch-concepts-demo/

Step 2 — Start with Docker Compose

This folder has a docker-compose.yml that starts both Elasticsearch and Kibana together.

cd elasticsearch-concepts-demo
docker compose up -d

Wait about 30 seconds, then check both services are running:

docker compose ps
Expected output: Both elasticsearch and kibana services should show status Up.

Step 3 — Verify Elasticsearch is running

# Open in your browser OR run this curl command:
curl http://localhost:9200

# You should see JSON like:
{
  "name": "...",
  "cluster_name": "talent-tracker-cluster",
  "version": { "number": "8.x.x", ... },
  "tagline": "You Know, for Search"
}

Step 4 — Open Kibana

Kibana is a web UI for Elasticsearch. Open your browser and go to:

http://localhost:5601
Kibana Dev Tools is what we use to run queries. Inside Kibana, go to:
Menu (hamburger icon) → Management → Dev Tools

To stop everything when you're done:

docker compose down
Note: Running docker compose down -v also deletes all indexed data. Use plain docker compose down to keep your data between sessions.
Data Indexing

5. Index Candidate Data from MongoDB into Elasticsearch

We have a Node.js script (scripts/index-candidates.js) that:

  1. Connects to your local MongoDB (where Talent Tracker data lives)
  2. Reads all documents from the candidates collection
  3. Creates the candidates index in Elasticsearch with the correct mapping
  4. Sends all candidate documents to Elasticsearch (bulk indexing)

Install dependencies first:

cd elasticsearch-concepts-demo
npm install

Run the indexing script:

node scripts/index-candidates.js

Expected output:

Connected to MongoDB
Connected to Elasticsearch
Creating index "candidates" with mapping...
Index created successfully
Fetched 5 candidates from MongoDB
Indexed 5 documents into Elasticsearch
Done!
How bulk indexing works: Instead of sending one document at a time, the bulk API sends all documents in one HTTP request — much faster for large datasets.

What the script does internally (simplified):

// 1. Read from MongoDB
const candidates = await Candidate.find({});

// 2. Build bulk request body
const body = candidates.flatMap(c => [
  { index: { _index: "candidates", _id: c._id.toString() } },
  {
    name: c.name,
    email: c.email,
    skills: c.skills,
    location: c.location,
    experience_years: c.experience_years,
    status: c.status
  }
]);

// 3. Send to Elasticsearch
await esClient.bulk({ body });
Querying

6. Run Basic Search Queries via Kibana Dev Tools

Open Kibana at http://localhost:5601, go to Dev Tools, and paste these queries one by one. Press the green play button (▶) to run each query.

Query 1 — Check how many documents are in the index

GET /candidates/_count

Query 2 — See all candidates (up to 10)

GET /candidates/_search
{
  "query": {
    "match_all": {}
  }
}

Query 3 — Full-text search by skill

Find candidates who have "React" in their skills (case-insensitive, partial match):

GET /candidates/_search
{
  "query": {
    "match": {
      "skills": "React"
    }
  }
}

Query 4 — Exact filter by location

Find candidates who are in "New York" exactly:

GET /candidates/_search
{
  "query": {
    "term": {
      "location": "New York"
    }
  }
}

Query 5 — Range query on experience

Find candidates with 3 or more years of experience:

GET /candidates/_search
{
  "query": {
    "range": {
      "experience_years": {
        "gte": 3
      }
    }
  }
}

Query 6 — Combine search + filter (bool query)

Find active candidates who know React AND have 2+ years experience — this is the most common pattern:

GET /candidates/_search
{
  "query": {
    "bool": {
      "must": [
        { "match": { "skills": "React" } }
      ],
      "filter": [
        { "term":  { "status": "active" } },
        { "range": { "experience_years": { "gte": 2 } } }
      ]
    }
  }
}
bool query explained:
must — affects relevance score (like WHERE clause that also ranks)
filter — yes/no gate, does NOT affect score (faster, cached)
should — optional boost (nice to have)
must_not — exclude results

Query 7 — Aggregation: count candidates by location

Like GROUP BY location in SQL — how many candidates per city?

GET /candidates/_search
{
  "size": 0,
  "aggs": {
    "by_location": {
      "terms": {
        "field": "location"
      }
    }
  }
}

Query 8 — Aggregation: average experience years

GET /candidates/_search
{
  "size": 0,
  "aggs": {
    "avg_experience": {
      "avg": {
        "field": "experience_years"
      }
    }
  }
}

Query 9 — Delete and re-create the index (reset)

# Delete the index
DELETE /candidates

# Verify it's gone
GET /candidates/_count
Warning: DELETE /candidates removes ALL indexed data. Only run this if you want to start fresh and re-run the indexing script.
